# Attendance Service Architecture

## Overview

The **Attendance Service** is a Spring Boot 3.2.2 microservice (Java 17) that provides generic attendance register management and event logging within the DIGIT-Works platform. It is responsible for:

- Creating and managing attendance registers
- Recording attendance events (IN/OUT timestamps) for individuals
- Tracking staff and attendee enrollment/de-enrollment
- Period-aware status management for billing integration (Payments V2)

The service follows event-driven architecture using Kafka for asynchronous persistence and integrates with multiple DIGIT backbone services.

## Technology Stack

- **Framework:** Spring Boot 3.2.2
- **Language:** Java 17
- **Database:** PostgreSQL 12 (schema-based multi-tenancy)
- **Persistence:** JDBC with custom RowMapper pattern (not JPA)
- **Message Queue:** Apache Kafka (async persistence and events)
- **ID Generation:** egov-idgen service
- **Database Migrations:** Flyway
- **Configuration Management:** MDMS (Master Data Management Service)
- **Caching:** Redis (with TTL support)

## Data Model

### Core Tables

#### 1. `eg_wms_attendance_register`
Primary entity representing an attendance register/batch
- `id`: Unique identifier
- `tenantid`: Tenant identifier for multi-tenancy
- `registernumber`: Unique register number (generated by idgen service)
- `name`: Register name/description
- `startdate`: Register start date (epoch timestamp)
- `enddate`: Register end date (epoch timestamp)
- `status`: Register status (e.g., CREATED, APPROVED, SUBMITTED)
- `reviewStatus`: Legacy status field for backward compatibility
- `periodStatuses`: JSONB field storing per-period muster roll statuses
- `referenceId`: Reference to external entity
- `serviceCode`: Service code from MDMS
- `localityCode`: Locality/boundary code
- `additionalDetails`: JSONB for extensible metadata
- **Audit Fields:** `createdby`, `lastmodifiedby`, `createdtime`, `lastmodifiedtime`

#### 2. `eg_wms_attendance_staff`
Staff members enrolled in attendance registers
- `id`: Unique identifier
- `register_id`: FK to `eg_wms_attendance_register`
- `individual_id`: Individual/user identifier
- `enrollment_date`: Date staff enrolled (epoch timestamp)
- `deenrollment_date`: Date staff de-enrolled (epoch timestamp)
- `stafftype`: Type of staff (e.g., HEALTH_WORKER, SUPERVISOR)
- `additionalDetails`: JSONB metadata
- **Audit Fields:** Standard audit columns

#### 3. `eg_wms_attendance_attendee`
Attendees/participants in a register
- `id`: Unique identifier
- `register_id`: FK to `eg_wms_attendance_register`
- `individual_id`: Individual/user identifier
- `enrollment_date`: Enrollment date (epoch timestamp)
- `deenrollment_date`: De-enrollment date (epoch timestamp)
- `tag`: Attendee classification/tag
- `additionalDetails`: JSONB metadata
- **Audit Fields:** Standard audit columns

#### 4. `eg_wms_attendance_log`
Individual attendance event logs
- `id`: Unique identifier
- `register_id`: FK to `eg_wms_attendance_register`
- `individual_id`: Individual identifier
- `status`: Attendance status (e.g., PRESENT, ABSENT)
- `time`: Event timestamp (epoch)
- `event_type`: Type of event (IN, OUT, etc.)
- `clientreferenceid`: Client-provided reference
- `documentIds`: Associated documents
- **Client Audit Fields:** `clientcreatedby`, `clientlastmodifiedby`, `clientcreatedtime`, `clientlastmodifiedtime`
- **Server Audit Fields:** Standard server audit columns
- `additionalDetails`: JSONB metadata

#### 5. `eg_wms_attendance_document`
Documents attached to attendance logs
- `id`: Unique identifier
- `attendance_log_id`: FK to `eg_wms_attendance_log`
- `filestore_id`: File storage reference
- `document_type`: Type of document
- `status`: Document status
- `additionalDetails`: JSONB metadata
- **Audit Fields:** Standard audit columns

#### 6. `eg_wms_staff_permissions` (Legacy)
Permission management for staff members (may be deprecated)
- `id`: Unique identifier
- `register_id`: FK to `eg_wms_attendance_register`
- `staff_id`: FK to `eg_wms_attendance_staff`
- `permission_type`: Type of permission
- `additionalDetails`: JSONB metadata
- **Audit Fields:** Standard audit columns

### Key Components

#### 1. REST Controllers (Request/Response)
- **AttendanceApiController** (`/v1/`)
  - `POST /_create`: Create attendance register
  - `POST /_update`: Update register
  - `POST /_search`: Search registers with filters

- **AttendanceLogApiController** (`/v1/`)
  - `POST /_create`: Record attendance event (IN/OUT)
  - `POST /_update`: Update attendance log
  - `POST /_search`: Search attendance logs

- **StaffApiController** (`/v1/`)
  - `POST /_create`: Enroll staff to register
  - `POST /_update`: Update staff enrollment
  - `POST /_search`: Search staff members

- **AttendeeApiController** (`/v1/`)
  - `POST /_create`: Enroll attendee
  - `POST /_update`: Update attendee
  - `POST /_search`: Search attendees

#### 2. Service Layer (Business Logic)
Services implement core business logic including:
- **Validation:** Request validation using validator classes
- **Enrichment:** Data enrichment (ID generation, timestamp assignments, etc.)
- **Persistence:** Event publication to Kafka topics
- **Search:** Complex query building with filters and pagination

#### 3. Repository Layer (Data Access)
- **JDBC-based:** Direct JDBC queries with custom RowMapper pattern
- **QueryBuilders:** Dynamic SQL construction for flexible filtering
- **ServiceRequestRepository:** HTTP client for external service calls

#### 4. Kafka Integration

##### Producer Topics
The service publishes to these topics:
- `save-attendance`: New attendance register creation
- `update-attendance`: Attendance register updates
- `save-attendance-log`: New attendance event logs
- `update-attendance-log`: Update attendance logs
- `save-staff`: Staff enrollment
- `update-staff`: Staff updates
- `save-attendee`: Attendee enrollment
- `update-attendee`: Attendee updates
- `mz-save-*` / `mz-update-*`: Variants for multi-schema (Mozambique) deployment

##### Consumer Topics
- `muster-roll-status-update`: Consumes muster roll workflow status updates (V2 Payments)
- `organisation.contact.details.update`: Organization staff contact info updates
- `contracts-revision`: Contract revision updates (for time extension)
- Health attendance bulk create/update topics for bulk operations

#### 5. Data Enrichment
Enrichment services add/validate data before persistence:
- Generate unique IDs using idgen service
- Add audit details (createdBy, createdTime, etc.)
- Validate references (register, individual IDs)
- Calculate derived fields (periodStatuses for V2)

## API Specifications

### Attendance Register Endpoints

**POST /attendance/v1/_create**
- Create a new attendance register
- Requires: `AttendanceRegisterRequest` with staff/attendee lists
- Returns: `AttendanceRegisterResponse`

**POST /attendance/v1/_update**
- Update existing register details
- Returns: Updated register response

**POST /attendance/v1/_search**
- Search registers with filters
- **V1 Parameters:** `tenantId`, `id`, `registerNumber`, `status`, `offset`, `limit`
- **V2 Parameters (Payments Integration):**
  - `billingPeriodId`: Filter by billing period
  - `registerPeriodStatus`: APPROVED or PENDING (muster roll status)
  - Returns `periodStatuses`: Per-period status map and derived `registerPeriodStatus`

### Attendance Log Endpoints

**POST /attendance/v1/_create**
- Record IN/OUT timestamps for individuals
- Supports document attachment (photos, etc.)

**POST /attendance/v1/_search**
- Search attendance logs with date range, status filters

### Search Features

All search endpoints support:
- **Pagination:** `offset` and `limit` parameters
- **Sorting:** On register number, date fields
- **Multi-tenancy:** Tenant-scoped results
- **Role-based Access:** Configurable open search roles (`attendance.register.open.search.enabled.roles`)

## Persister Configuration

The service uses a persister YAML file (`attendance-service-persister.yml`) that defines Kafka-to-database mappings:

### Service Maps
- **Attendance Register:** Persists `eg_wms_attendance_register` + nested staff
- **Attendee:** Persists `eg_wms_attendance_attendee` records
- **Staff:** Persists `eg_wms_attendance_staff` records
- **Attendance Log:** Persists `eg_wms_attendance_log` + documents

Each mapping includes:
- **Topic:** Kafka topic to consume from
- **Query:** SQL INSERT/UPDATE template
- **JsonMaps:** Field mappings from request JSON to SQL parameters

### Multi-Schema Support
Separate topic mappings for:
- `public` schema (default)
- `mz` schema (Mozambique tenant)

## V2 Payments Integration (Intermediate Billing)

### Period Status Tracking
The service tracks muster roll statuses per billing period:

- **Muster Status Updates:** Consumed via `muster-roll-status-update` Kafka topic
- **Storage:** JSONB `periodStatuses` field in `eg_wms_attendance_register`
- **Search Parameters:**
  - `billingPeriodId`: Filter registers overlapping a period
  - `registerPeriodStatus`: Search by APPROVED/PENDING status

### Billing Period Lookup
- Fetches billing periods from **Expense Calculator Service**
- Endpoint: `GET /health-expense-calculator/billing-config/v1/periods/_search`
- Caches period metadata locally

### Muster Roll Status Fallback
- If period status not in local `periodStatuses`, queries **Muster Roll Service**
- Endpoint: `GET /health-muster-roll/v1/_search`
- Async sync via Kafka ensures eventual consistency

## Multi-Tenancy

The service supports multi-tenancy through:
- **Schema-based:** Separate PostgreSQL schemas per tenant
- **Configuration:** `state.level.tenantid.length` (typical: 1 char for state code)
- **Request Scoping:** All API calls include `tenantId` in request
- **Data Isolation:** Queries filtered by tenant automatically

### Multi-State Instance (Central)
- Configuration: `is.environment.central.instance`
- For central deployments managing multiple state instances
- Uses `MultiStateInstanceUtil` for tenant routing
